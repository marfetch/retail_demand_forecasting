# Stationarity Report

Актуализировано по файлу `data/processed/daily_sales_series.csv` после повторного запуска ноутбуков: `2026-05-02`.

## Зачем проверять стационарность

Для ARIMA/SARIMA важно понимать, является ли временной ряд стационарным. Нестационарный ряд может иметь тренд, меняющуюся дисперсию или сезонность, из-за чего модель хуже оценивает зависимости во времени. Проверка стационарности помогает обосновать дифференцирование и сезонную компоненту.

## Декомпозиция ряда

В проекте используется декомпозиция дневного ряда с периодом `7`, чтобы выделить недельную сезонность:

- trend;
- seasonal;
- residual.

График:

`images/decomposition.png`

Декомпозиция показывает недельную сезонность и меняющийся уровень ряда. Это согласуется с EDA: продажи заметно отличаются по дням недели.

## ADF-тест исходного ряда

ADF-тест проверяет нулевую гипотезу о наличии unit root, то есть о нестационарности ряда.

p-value исходного ряда:

`0.0036`

Так как `p-value < 0.05`, есть основания считать исходный ряд стационарным по ADF-тесту на уровне значимости 5%. При этом визуальная сезонность остается, поэтому сезонная компонента в SARIMA все равно уместна.

## Логарифмирование

Для стабилизации масштаба используется преобразование `log1p(sales_clean)`. Оно снижает влияние крупных значений и делает относительные изменения более сопоставимыми.

## Дифференцирование

Для дополнительной проверки используется первое дифференцирование логарифмированного ряда. Это стандартный прием для удаления локального тренда и подготовки ряда к ARIMA-подходу.

## ADF-тест после преобразования

p-value после преобразования:

`1.32e-28`

После логарифмирования и первого дифференцирования ряд также стационарен по ADF-тесту.

## Вывод для ARIMA/SARIMA

Несмотря на то что ADF-тест исходного ряда уже дает `p-value < 0.05`, в проекте используется SARIMA с `order=(1, 1, 1)` и `seasonal_order=(1, 1, 1, 7)`, потому что ряд демонстрирует недельную сезонность, а дифференцирование помогает модели устойчивее работать с изменениями уровня продаж.
